# Design System: MHWs Equipment Manager

## デザイン指針
視認性と操作性を両立するため、データ密度の高いコンパクトなビジュアルと識別しやすいカラーリングを標準とする。また、AIエージェントによる長期的なメンテナンスを想定し、関心事の物理的な分離を最優先する。

## 物理アーキテクチャ設計 (AI-Friendly Design)
AIエージェントが最小限のトークンで確実な修正を行えるよう、ページを以下の 5 つの役割に分解して配置する。

- **`state.py` (State Layer)**: Session State の初期化と同期ロジック。
- **`atoms.py` (Atomic Visual Layer)**: 
  - デザイン修正（カラー、アイコン、バッジ等）の唯一の修正者。
  - ロジックを持たない純粋な表示パーツのみを定義する。
- **`dialogs.py` (Interaction Layer)**: `@st.dialog` を用いた浮動型UI。
- **`form.py` (Input Layer)**: 検索、フィルタ、登録等の入力フロー。
- **`list.py` (Data Layer)**: リストやグリッド等のデータ描画。

## 各コンポーネントのデザイン標準
- **超スリム・シングル行カード**: 
  - 1レコード1行。全画面で「ウルトラスリム」なカード形式を採用し、高密度での情報一覧性を確保する。
- **属性別カラーバッジ**: 
  - 属性名称のみ（火, 水, 雷...）を表示する単体バッジ。背景色はモンハン標準の属性イメージカラー（MHカラー）を適用する。

## アクセシビリティ・レイアウト
- 余白を極力減らし、固定幅カラムを多用して位置ズレを防ぐ。
  - 特にリスト形式（HUD v14）では、最終項目の「数値/回数」を揃えるため、中間のテキストエリアに厳格な固定幅を適用する。
- Undo/Redoなどの操作系ボタンはツールバーに並べて配置する。

## パフォーマンスと応答性
- **ゼロ・ディスクI/O原則**: ページ内でのI/Oは最小限とし、マスタデータ等は初回読み込み後にメモリキャッシュ（`st.cache_resource` / `st.cache_data`）で保持する。
- **入力優先度**: ユーザーのテキスト入力（Session State）に対する応答を最優先し、バックグラウンドでの重いリスト生成が入力のブロッキングを起こさないよう `st.fragment` で非同期的に処理する。

## 状態遷移と比較のビジュアル標準 (Visual Band)
情報の「変化（Before/After）」を示す際は、以下のビジュアルルールを適用する。

- **Before（現在）**: 
    - 左端に `2px` のグレー (`#666`) バーを表示。
    - `font-size: 0.85rem`、不透明度 `0.6` 程度で muted に表示。
- **After（目標）**: 
    - 左端に `4px` の鮮やかな緑（`#2ecc71` などの成功カラー）バーを表示。
    - **`font-size: 1.2rem`** 程度まで拡大し、`font-weight: bold` で目標状態を強調。
- **整列要件**: データのズレを防ぐため、常に固定幅グリッド（例：5カラム等幅）内で表示し、文字列の長さによるレイアウト崩れを許容しない。
