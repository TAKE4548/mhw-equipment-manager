# UI Specification: MHWs Equipment Manager

## 画面構造とナビゲーション
- **ナビゲーション構造**:
  - **Home (🏠)**: Altairを使用した統計分析ダッシュボード。武器種・属性・スキルの分布を可視化し、ドリルダウン分析が可能。
  - **所有巨戟アーティア一覧 (📦)**: 所持武器の登録・編集・削除。超スリムカードデザインを採用。
  - **スキル抽選結果 (⚔️)**: 武器種・属性ごとのスキル抽選シードのトラッキング管理。
  - **復元強化厳選 (✨)**: 武器個別の復元ボーナステーブル進行管理。強化前後の比較ビューを備える。
  - **鑑定護石一覧 (📿)**: *[Completed]* ランダム護石の登録（カスケード制御）、スキル・スロット検索、お気に入り管理。
    - **検索・フィルタ機能 (🔍)**: [NEW] 膨大な護石の中から目的のものを探すため、レア度、スキル名、最低スロットレベルによる多角的なフィルタリングが可能。
    - **1クリック・レスポンス (UX)**: `st.fragment` 内のウィジェットに対し `key` 同期を使用し、操作が1回で即座に反映されることを保証。
    - **ソート機能**: レア度高順、スキル名順、お気に入り優先などの切り替えが可能。

- **共通設定サイドバー**:
  - 全ページで同一の「ストレージ設定」（ローカル保存 / クラウド同期）とログイン機能にアクセス可能。

## コンポーネント役割と表示ロジック
- **統合型登録フォーム**:
  - 新規スキル抽選結果の登録は、メイン画面トップのエキスパンダー内で行う（ページ遷移を最小化）。

- **ビジュアル武器選択 (Premium Visual Selection / HUD v14)**: 
  - 抽選結果の登録や付与の際、詳細情報を含んだ「プレミアム・カード一覧」から視覚的に武器を選択可能。
  - **統一アクションボタン方式**: 武器情報を表示するタグカードの右端に「❯ / ✔」のアクションボタンを配置し、統合レイアウト比率（11.5:1）による垂直同期を保証。
  - **視覚的フィードバック**: 選択時にはゴールド発光ボーダーを適用。
  - **高機能フィルタ (REQ-072)**: 
    - **3カラム・グリッドレイアウト**: 情報を「基本(L1)」「ビルド(L2)」「状態(L3)」の3軸に整理して配置。
    - **スキル・マルチ検索**: シリーズスキル、グループスキルをそれぞれ独立したマルチセレクトで検索可能。
    - **ボーナス種別フィルタ (AND検索)**: 
      - 「種類+レベル」単位の一致検索。
      - 複数選択した場合、その**すべての項目**を復元ボーナス枠（5枠）内に持つ武器を抽出する。
      - 生産ボーナスは検索対象から除外（復元厳選の効率化に特化）。
  - **ダイアログ内2段階選択 (鑑定護石専用)**: スキル検索性を高めるため、
「スキル名選択」→「レベル選択」を同一ダイアログ（st.dialog）内で画面を閉じずに遷移させるUIを採用しています。
    - **オート・コンファーム (UX向上)**: スキル名を選択した際、有効なレベルが1種類のみであれば、第2段階（レベル選択）をスキップし、値を自動確定してダイアログを閉じます。これにより操作コストを最小化します。

- **強化前後の比較バー (HUD Micro-Diff / v14)**: 
  - 5つの枠（ボーナス）を一塊のセットとして捉え、現状と抽選結果のボーナスの推移（before ❯❯ after）を、テーブルではなく「色の帯（バー）」と「輝度の差」を模した 1 行の HTML 要素で表現します。
  - 各項目を等幅グリッドで配置し、文字数に関わらない完璧な上下整列を実現します。
  - 変化がある要素を強調（ゴールド色への変化による Diff 表示）。

- **プログレッシブ入力制御 (Progressive Disclosure)**:
  - フォームにおいて、入力に前後関係や依存関係（例: レア度→第1スキル→第2スキル、あるいは枠1→枠2）がある場合、前段の入力が完了するまで後段の要素を `disabled` (無効化) 状態とし、左から右へ自然と順番に誘導する設計を取り入れます。
  - **依存関係のリセット**: レア度や上流のスキルが変更された場合、整合性を保つためそれ以降の全項目の選択状態を強制的にリセット（初期化）します。これにより常に「正しい組み合わせ」のみが入力される状態を保証します。

## パフォーマンス・描画アーキテクチャ
高密度なデータを高速に描画し、入力ラグを最小化するため、以下の構成を標準とします。

- **フラグメント隔離 (st.fragment)**:
  - 「登録フォーム」と「一覧表示リスト」は必ず別の `st.fragment` 単位で隔離します。これにより、フォームへの1文字入力（Session State更新）が、重いリスト描画の再トリガーにならないように制御します。
  - **遵守事項（不整合防止）**:
    1. **グローバル同期の強制**: リスト内での削除や更新によってページ全体の再描画が必要な場合（例：ヘッダーの統計値が変わるなど）、明示的な `st.rerun()` を呼び出してページ全体の整合性を保つこと。
    2. **データの共通参照**: フラグメント外で読み込んだデータを引数として渡す際、そのデータが最新であることを保証するため、I/O層（load_* 関数）のキャッシュクリアタイミングに注意すること。
    3. **Session State の競合回避**: フラグメント内でのみ使用する一時的な状態（例：検索フィルタの途中経過）は、キーにユニークなプレフィックスを付けるか、意図しないリセットが起きないよう設計すること。
- **事前データ加工 (Pre-render Preparation)**:
  - 描画ループ（`for` 文）内での `merge`, `normalize`, `load_data` 呼び出しを禁止します。
  - ループに入る前に必要なデータをすべて結合・加工済みの DataFrame として用意し、ループ内では単純な参照のみを行う構成にします。
- **ロジック層のキャッシュ化**:
  - `master_data.json` 等の静的データ読み込み、および正規表現を用いた略称変換（`get_abbr_item` 等）には必ず適したキャッシュ（`st.cache_data` または `lru_cache`）を適用します。

## 表示用語の統一 (Terminology)
- **ボーナス名称**: 
  - 生産: `基礎攻撃力増強`, `会心率増強`
  - 復元（厳選対象）: `基礎攻撃力強化`, `会心率強化`, `属性強化`, `切れ味強化`, `装填強化`
- **レベル表記**: 
  - ローマ数字（`Ⅰ`, `Ⅱ`, `Ⅲ`）および `EX`, `無印` を使用。
