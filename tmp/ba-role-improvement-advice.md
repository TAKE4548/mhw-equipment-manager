# BA ロール・要求分析フェーズ改善アドバイス

> 実運用で発覚した「要求分析が浅い」問題への対策。
> BA ロール定義と requirement-analysis スキルの見直し時に参照する。

---

## 1. 発覚した問題

### 症状

BA が要望を素直にバックログにストックし、対処療法的なやり取りになりがち。
ユーザーの発言を額面通りに受け取り、掘り下げが行われない。

### 実例

```
ユーザー: 「カードを一覧表示したとき、中の同じ項目の位置がずれてるのをキレイにそろえたい」
  ↓
BA が登録した要求: 「カード内要素の位置を揃える」
  ↓
Architect の設計: 固定幅で位置合わせ
  ↓
実装計画の確認で判明: 他の項目が短縮されて見づらくなる
  ↓
ユーザーの追加指示: 「項目の順番入替込みで調整できないか」
  ↓
ようやくより良い案が出てくる
```

### 本来あるべき姿

要求分析の時点で「位置を揃える」ではなく、
「カード一覧で項目間の比較が直感的にしやすい UI にしたい」という真の要求に落とし、
設計フェーズに入るべきだった。
そうすれば Architect は固定幅以外の選択肢（項目順の入替、グリッドレイアウト、
テーブル表示への切替など）も最初から検討できた。

---

## 2. 根本原因

### BA が「記録係」になっている

現行の BA ロールは「ユーザーの発言を構造化してバックログに入れる」仕事をしているが、
本来は「ユーザーの言葉を設計可能な要求に翻訳する」仕事をすべき。

**記録係**: 発言 → 整理 → バックログ登録（今の状態）
**翻訳者**: 発言 → 掘り下げ → 真の課題特定 → 要求定義 → バックログ登録（あるべき姿）

### スキル定義に「掘り下げの方法論」がない

今の requirement-analysis スキルの Steps:

```
1. ユーザの発言を原文のまま記録
2. 発言から「課題」を抽出
3. 課題から「要求」を導出
```

手順としては書いてあるが、**どう掘り下げるかが定義されていない**。
「課題を抽出」「要求を導出」だけでは、発言をそのまま言い換えて終わってしまう。

### ユーザーの発言は「解決策」か「症状」であることが多い

ユーザーは自分なりに原因を考えた上で発言するため、
発言の中に具体的な手段が混ざっていることが多い。

- 「位置を揃えたい」← 手段
- 「色を変えたい」← 手段
- 「ボタンを大きくしたい」← 手段
- 「別画面に分けたい」← 手段

これらをそのまま要求として登録すると、Architect の設計の自由度を狭めてしまう。

---

## 3. 改善方針

### BA ロールの Mindset に追加すべき観点

#### 掘り下げの姿勢

- ユーザーの発言は「解決策」や「症状」であることが多いと常に意識する
- 発言をそのまま要求にしない。必ず「なぜそれが必要か」を掘り下げる
- 「それによって何を達成したいか」を確認する
- ユーザーが具体的な手段に言及している場合は特に注意する

#### 掘り下げのフレーム: Surface → Symptom → Root Cause → Requirement

4段階で発言を分解する。

| 段階 | 問い | 今回の例 |
|------|------|---------|
| Surface（発言） | ユーザーは何と言ったか | 「位置がずれてるのを揃えたい」 |
| Symptom（症状） | 何が実際に困っているか | カード一覧で同じ項目の位置がカードごとにバラバラ |
| Root Cause（真の課題） | なぜそれが困るか | 一覧表示時に項目間の比較が直感的にしにくい |
| Requirement（要求） | 何が満たされれば解決するか | カード一覧で項目間の比較が直感的にできる UI にする |

**Surface と Requirement が同じ抽象度になっていたら掘り下げが足りない。**
「位置を揃えたい」→「位置を揃える」は言い換えであって分析ではない。

#### 「手段」と「目的」の判定

ユーザーの発言が以下のパターンの場合、手段に言及している可能性が高い:

- 「xxxしたい」「xxxを直したい」「xxxに変えたい」（具体的な操作・変更に言及）
- 「xxxみたいにしたい」（他サービスの具体的な実装に言及）
- 「xxxを追加してほしい」（具体的な機能・要素に言及）

これらを受けたとき、BA はその手段が達成しようとしている目的を探る。

#### 掘り下げの質問テンプレート

Surface に手段が含まれている場合の質問パターン:

```
「{発言内容}とのことですが、それは{推測される目的}ということでしょうか？
 他にも気になっている点があれば教えてください。」
```

例:
```
「位置を揃えたいとのことですが、それはカードを一覧で見たときに
 項目同士を比較しやすくしたい、ということでしょうか？
 他にも見づらいと感じている点があれば教えてください。」
```

この質問で:
- 推測が合っていれば、Root Cause が確認できる
- 推測が外れていれば、ユーザーが本当の目的を教えてくれる
- 「他にも」で追加の症状が出てくることが多い

#### 関連する症状の探索

1つの Root Cause から複数の Symptom が出ていることがある。
ユーザーが言及した症状だけでなく、同じ課題から来ている他の症状がないか探る。

今回の例で言えば:
- 位置がずれている（ユーザーが言及した症状）
- フォントサイズや表示形式がカードごとに微妙に違う（言及されていない可能性のある症状）
- そもそもカードのレイアウト自体が比較に適していない（より根本的な症状）

これらを洗い出すことで、Architect が設計時に考慮できる範囲が広がる。

## 4. requirement-analysis スキルの Steps 改修案

```
1. ユーザの発言を原文のまま記録（Surface）

2. 発言が「手段」か「目的」かを判定する
   - 具体的な実装に言及 → 手段の可能性が高い
   - 困りごとや体験に言及 → 目的に近い

3. 手段が含まれている場合、目的を掘り下げる
   - ユーザーに「それによって何を達成したいか」を確認する
   - 推測を提示して確認する形が有効

4. 症状（Symptom）を特定する
   - ユーザーが実際に体験している困りごとは何か
   - 同じ課題から来ている他の症状がないか探る

5. 真の課題（Root Cause）を特定する
   - その症状はなぜ困るのか
   - 「なぜ」を1〜2回掘り下げれば十分（掘りすぎると抽象的になりすぎる）

6. 要求を「目的」レベルで定義する
   - 具体的な手段ではなく、達成したい状態で記述する
   - 手段の検討は Architect が設計時に行う範囲

7. 受入基準を「目的」レベルで定義する
   - ✅「一覧表示で同じ項目同士を視線移動なく比較できる」
   - ❌「項目のX座標が全カードで一致する」（これは設計・実装の話）
   - 受入基準に具体的な実装手段が混ざっていたら定義し直す

8. docs/backlog.md に追記
```

### 掘り下げすぎの防止

「なぜ」を無限に繰り返すと、最終的に「ユーザー体験を良くしたい」のような
抽象的すぎる要求に行き着いてしまう。これでは設計の指針にならない。

掘り下げの適切な深さの目安:
- Architect がその要求から複数の設計選択肢を検討できるレベル
- 具体的な手段が1つに限定されないが、方向性は明確なレベル

```
浅すぎ: 「位置を揃える」（手段が1つに限定される）
適切:   「一覧で比較が直感的にできるUIにする」（方向性は明確、手段は複数ありえる）
深すぎ: 「使いやすいアプリにする」（何でもありすぎて指針にならない）
```

## 5. バックログ項目フォーマットの拡張案

現行:
```
Source / Problem / Requirement / Acceptance criteria
```

改修案:
```
Surface:          ユーザーの原文
Symptom:          実際に体験している困りごと
Root Cause:       なぜそれが困るか
Requirement:      目的レベルの要求
Acceptance criteria: 目的レベルの受入基準
```

今回の事例で書くとこうなる:

```
### REQ-12: カード一覧の比較性改善
- Surface: 「カード内の同じ項目の位置がずれてるのを揃えたい」
- Symptom: カード一覧で同じ項目の位置がカードごとにバラバラ
- Root Cause: 一覧表示時に項目間の比較が直感的にしにくい
- Requirement: カード一覧で項目間の比較が直感的にできるUIにする
- Acceptance criteria:
  - 一覧表示で同じ項目を視線移動なく比較できる
  - 既存の項目の可読性が損なわれない
  - カード内の情報量が減らない
```

この形なら Architect は「比較しやすさ」を達成する手段を
設計段階で幅広く探ることができる。

---

## 6. BA ロールの Boundaries の見直し

現行の Boundaries に以下の観点を追加する:

```
- ユーザーの発言をそのまま要求として登録しない
- Surface と Requirement が同じ抽象度の場合は掘り下げが不足している
- 要求に具体的な実装手段が含まれている場合は定義し直す
- 受入基準に実装手段が混ざっていないか確認する
- 手段の検討は Architect の領域であり、BA は目的の定義に留める
```

---

## 7. Architect 側への影響

BA が目的レベルの要求を出すようになると、Architect は設計の自由度が上がる一方で、
「複数の実現手段を比較検討して最適な設計を選ぶ」責務が明確になる。

system-design スキルの Steps に以下の観点を追加することを検討:

```
- 要求（目的）に対して複数の実現手段を列挙する
- 各手段のトレードオフ（メリット・デメリット・影響範囲）を明示する
- 推奨案とその理由を提示する
- ユーザーが判断できる形で選択肢を提示する
```

これにより「固定幅で揃える」一択ではなく、
「固定幅 / 項目順入替 / グリッドレイアウト / テーブル表示」の比較表が出てくる。

---

## 8. 改善効果の検証方法

BA の改善が効いているかを判断する指標:

- **設計フェーズでの手戻り減少**: Architect の設計に対してユーザーが「そうじゃなくて」と言う頻度が減るか
- **要求と受入基準の抽象度**: 具体的な手段ではなく目的レベルで書かれているか
- **Architect の設計選択肢**: 設計ドキュメントに複数の実現手段が列挙されるようになるか
- **差し戻しの原因変化**: 「要求の認識齟齬」による差し戻しが減るか

## 9. new / ready ステータスの分離

### なぜ分離が必要か

掘り下げプロセスを導入すると、「BA が登録したけどまだ掘り下げが終わってない」状態が
明確に発生する。今までは new → ready の境界が曖昧で、
実質「BA が登録したらすぐ ready」になっていたが、
これだと掘り下げが不十分なまま /dev に入れてしまう。

### ステータスの再定義

| ステータス | 意味 | 操作者 | /dev の対象 |
|-----------|------|--------|------------|
| new | BA が登録済み。掘り下げ未完了、またはユーザーとの整合がまだ | BA + ユーザー | ❌ |
| ready | 掘り下げ完了。Root Cause と目的レベルの要求が確定。開発着手可能 | BA が変更 | ✅ |

### new → ready の遷移条件

BA が以下をすべて確認して ready に変更する:

- Surface → Symptom → Root Cause → Requirement の 4 段階が埋まっている
- Requirement が「目的レベル」で書かれている（具体的な手段ではない）
- Acceptance criteria が「目的レベル」で書かれている
- ユーザーが要求の内容を確認している

**チェック方法**: Surface と Requirement を並べて読み、
同じ抽象度になっていたら ready にしない。

### Intake フローへの影響

掘り下げまで完了した場合:
```
「バックログに追加しました（REQ-{n}、ready）。
 /dev で開発に入れます。」
```

掘り下げが途中の場合（ユーザーが急いでいる、時間がない等）:
```
「バックログに仮登録しました（REQ-{n}、new）。
 まだ要求の整理が途中です。次回続きから整理できます。」
```

これにより、思いつきで投げた要望がそのまま開発に流れることを構造的に防ぐ。

### /dev Workflow への影響

Coordinator が Step 1 で backlog を提示するとき:

```
- ready / fix-needed / needs-investigation → 開発対象として選択可能
- new の項目がある場合は件数だけ通知:
  「未整理の要望が {n} 件あります。/dev の前に整理しますか？」
  - はい → BA ロールに切り替えて掘り下げ → ready にしてから /dev 継続
  - いいえ → ready 以降の項目だけで進める
```

### 掘り下げの再開

new のまま放置された項目を後日整理するケース:

- ユーザーが「バックログ整理したい」と発言 → BA ロール + backlog-management スキル
- または /dev の Step 1 で Coordinator が new 項目の整理を提案
- BA が前回の Surface を起点に掘り下げを再開
- 完了したら ready に変更

### backlog-management スキルへの影響

ステータス遷移ルールに以下を追加:

```
- new → ready: BA が掘り下げ完了を確認した場合のみ
- new のまま /dev の対象にしない
- Coordinator は new 項目を開発対象として提示しない
```

### 改善効果の検証への追加指標

- **new → ready の変換率**: 登録された要望のうち何割が ready まで到達するか
- **new のまま放置される件数**: 掘り下げが途中で止まっている項目の蓄積度合い
- **ready 項目の設計フェーズ通過率**: ready で入った項目が設計フェーズで手戻りなく通過するか

## 10. 実装時の注意：BA ロール定義の肥大化防止

### リスク

本ドキュメントの改善内容をすべて BA ロール定義（Rules）に詰め込むと、
記述量が膨らんで遵守性が落ちる可能性がある。
AntiGravity はワークフロー遵守の強制力が弱いため、指示が多いほど薄まりやすい。

### BA は他ロールより肥大化リスクが低い理由

Engineer はローレベル設計・テスト設計・TDD・ブラウザデバッグ・証跡作成と
やることが多岐にわたるため、Boundaries を膨らませると抜けが出やすい。

BA のやることは本質的に1つの流れに集約される:
```
ユーザーの発言を受けて → 掘り下げて → 要求として定義して → backlog に登録する
```

追加するUX観点も掘り下げフレームも、この流れの中の精度向上であって新しい仕事の追加ではない。
したがって改善内容自体は許容範囲。

### ただし Rules と Skill の分離を厳守する

すべてを Rules に入れるのではなく、**「何を意識するか」は Rules、「どうやるか」は Skill** に分離する。

**Rules（business-analyst.md）に入れるもの — 判断基準のみ、コンパクトに**:

```
- ユーザーの発言は手段や症状であることが多い。そのまま要求にしない
- Surface と Requirement が同じ抽象度なら掘り下げ不足
- Root Cause を特定するとき、UX 観点（情報設計 / インタラクション / 視覚）で分類する
- 手段の検討は Architect の領域、BA は目的の定義に留める
- new → ready の遷移は掘り下げ完了を確認してから
```

**Skill（requirement-analysis/SKILL.md）に入れるもの — 具体的な手順とテンプレート**:

```
- 掘り下げフレームの 4 段階（Surface → Symptom → Root Cause → Requirement）
- 手段 / 目的の判定パターンと例
- 掘り下げの質問テンプレート
- UX 観点での分類の具体的な判定基準と例
- 掘り下げ深さの目安（浅すぎ / 適切 / 深すぎの例）
- new → ready の遷移条件チェックリスト
- バックログ項目のフォーマット（Surface / Symptom / Root Cause / Requirement / Acceptance criteria）
```

### なぜこの分離が効くか

- Rules は model_decision トリガーで読み込まれるため、記述量がコンテキストを直接圧迫する。
  判断基準だけに絞ればコンパクトに収まる。
- Skill はセマンティックトリガーで発火したときだけ読み込まれるため、
  手順やテンプレートが多くてもコンテキストへの影響は限定的。
- BA が要望を受けて掘り下げる場面では Skill が発火するので、
  そのときだけ詳細な手順が使える。普段はコンテキストを消費しない。

### UX 観点の配置について

BA に追加する UX 観点は「これは UI/UX の問題だと気づく目」であり、
「UI を設計する能力」ではない。

- BA の UX 観点: Root Cause の分類（情報設計 / インタラクション / 視覚のどれか）
- UX デザイナーの能力: 分類された問題に対する具体的な UI 設計

BA が Root Cause を UX 観点で分類しておくと、
Architect → UX デザイナーへの切り替え判断が精確になり、
UX デザイナーも「何の問題を解決するか」が明確な状態で設計に入れる。

既存の UX デザイナー切り替えの仕組みは変更不要。
BA からの入力の質が上がることで、後段が自然に活きるようになる。
