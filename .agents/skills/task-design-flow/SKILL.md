# 🏛️ Design Workflow

### Phase 0: 要求の明文化
- `docs/requirements.md` に要求（REQ-XXX）を追記せよ。この記録なしに次へ進んではならない。

### Phase 1: 要求の構造化
- 解決すべき課題を整理し、ユーザーと論点を合意せよ。

### Phase 2: 設計書の改廃 (Single Source of Truth)
- 合意に基づき、`docs/` 内の関連ファイルを全て更新せよ。
- UIの変更がある場合は、必ず `design_system.md` と `ui_spec.md` の一貫性を確認し、競合があればこの段階で解消せよ。

### Phase 3: 実装・テスト計画書の作成 (implementation_plan.md)
- エンジニア向けに `implementation_plan.md` を作成し、以下を必須項目として含めよ：
  - **1. 概要**: 今回の改修の目的とゴール。
  - **2. 影響範囲**: 修正・新規作成が必要なファイルのフルパス一覧。
  - **3. ステップ別手順**: データの変更、ロジックの修正、UIの修正。
  - **4. テストシナリオ (最重要)**:
    - **正常系**: 期待される挙動と、確認すべきデータの状態。
    - **異常系**: 防ぐべきエラーと、その時のバリデーション挙動。
    - **ブラウザ操作**: `browser_subagent` 等で行う具体的な操作ステップ。
  - **5. 完了定義 (Definition of Done)**: 何をもってこのタスクを「完了」とするかの明示。
