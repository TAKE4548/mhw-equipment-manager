# 🛠️ Implementation Workflow (Mandatory)

> [!IMPORTANT]
> このフロー（Phase 0-4）は、承認済みの `implementation_plan.md` または明確な指示がある場合にのみ開始せよ。各フェーズの完了報告時には必ず証跡を添えよ。

### Phase 0: 仕様の同期とレビュー (Acceptance)
- `docs/` の最新仕様と「実装プラン」を熟読せよ。
- **【必須】** `docs/workflow/test_scenarios.md` を確認し、不足や不明点があれば直ちにアーキテクトに差し戻せ。

### Phase 1: 環境準備
- 実装用のGitブランチを作成し、必要に応じて仮想環境や依存関係を確認せよ。

### Phase 2: テストの先行作成 (Test-First)
- ロジック修正前にテストコードを作成するか、既存のテストを更新して期待される挙動を定義せよ。

### Phase 3: 実装と自律検証
- **【自律的差し戻し】** 設計や仕様に不備や実現困難な矛盾を発見した場合、直ちに実装を中断し、`### [Architect] task-design-flow: Phase 2 (差し戻し)` を宣言してアーキテクトとして設計の修正からやり直せ。
- `design_system.md` および `domain_rules.md` を厳守せよ。
- `browser_subagent` 等を活用し、実際の画面でテストシナリオ通りに動くか確認せよ。
- **検証報告**: 完了時、どのようなテストを行い、どのような結果が得られたか証跡を添えて記述せよ。

### Phase 4: 高品質なコミットと完了報告
- コミットプレフィックスを用いた、粒度の細かいコミットを実行せよ。
- アーキテクトが作成した `implementation_plan.md` 内のタスク（チェックリスト）を更新し、完了を宣言せよ。
